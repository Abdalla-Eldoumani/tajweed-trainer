# Content schema

Everything visible in the lessons and the practice quizzes comes from the JSON files in `src/data/content/`. This document explains the schema and how to add a new rule or example.

## Files

| File | What it holds |
|------|---------------|
| `learning-path.json` | Module ordering, prerequisites, descriptions |
| `makharij.json` | 5 articulation regions, 17 specific points, 29 letters |
| `noon-sakinah-tanween.json` | Izhar, Idgham, Iqlab, Ikhfaa rules |
| `meem-sakinah.json` | Ikhfaa Shafawi, Idgham Shafawi, Izhar Shafawi |
| `qalqalah.json` | 5 letters, 3 levels (Sughra, Wusta, Kubra) |
| `madd-rules.json` | 7 madd types, summary table |
| `laam-raa-rules.json` | Laam (sun / moon, lafzul jalalah), Raa (tafkheem / tarqeeq) |
| `tafkheem-tarqeeq.json` | Always-heavy, always-light, variable letters |
| `ghunnah.json` | Noon and Meem mushaddad, ghunnah prominence ranking |
| `waqf-symbols.json` | 11 stopping symbols and their effects |
| `glossary.json` | Tajweed terminology |
| `surah-index.json` | 114 surahs with `name_arabic`, page range, `bismillah_pre`, `revelation_place`. Generated by `scripts/fetch-surah-names.mjs`; do not hand-edit. |

## Common fields

Every entry that ships to the UI has a `verified: true` flag. Set it to `true` only after a domain reviewer has cross-checked the entry against primary sources (Al-Jazariyyah, the Mushaf, scholarly consensus).

For bilingual support, every visible English string has an optional `_ar` counterpart:

```jsonc
{
  "title_en": "Izhar (Clear Pronunciation)",
  "title_ar": "الإظهار الحلقي",
  "description": "When Noon Sakinah or Tanween is followed by ...",
  "description_ar": "إذا وقعت النون الساكنة أو التنوين قبل ...",
  "common_mistakes": ["..."],
  "common_mistakes_ar": ["..."]
}
```

Rendering picks the `_ar` field when `useTranslation().isAr` is true and the field is non-empty; otherwise it falls back to English. Don't hardcode Arabic strings in components — add them to JSON and let the rendering layer choose.

## TajweedRule

```ts
interface TajweedRule {
  id: string;            // unique within file
  order: number;         // display order
  title_en: string;
  title_ar: string;
  description: string;
  description_ar?: string;
  arabic_definition?: string;     // canonical Arabic phrasing
  letters?: ArabicLetter[];
  letters_description?: string;
  letters_description_ar?: string;
  subtypes?: TajweedSubtype[];
  examples: QuranicExample[];
  audio_tip?: string;
  audio_tip_ar?: string;
  common_mistakes?: string[];
  common_mistakes_ar?: string[];
  mnemonic_ar?: string;           // memory aid in Arabic
  mnemonic_en?: string;
  verified: true;
}
```

## QuranicExample

Every example must include exact references and a highlight word:

```ts
interface QuranicExample {
  arabic: string;            // full Arabic text with tashkeel
  transliteration: string;
  translation: string;
  translation_ar?: string;
  surah: number;             // 1..114
  ayah: number;
  surah_name_en: string;
  surah_name_ar?: string;    // populated by scripts/fetch-surah-names.mjs
  highlight_word: string;    // the word demonstrating the rule
  rule_applied: string;      // which sub-rule applies
  rule_applied_ar?: string;
}
```

Hard rules:

- `arabic` must be in Uthmani script with full tashkeel; a missing diacritic can change meaning.
- `surah` and `ayah` together identify the example uniquely. If either is wrong the example is wrong.
- `transliteration` follows ALA-LC: `ع = '` (ayn), `ء = '` (hamza), `ḥ` (ح), `kh` (خ), `dh` (ذ), `sh` (ش), `ṣ` (ص), `ḍ` (ض), `ṭ` (ط), `ẓ` (ظ), `gh` (غ), `q` (ق), `th` (ث).
- `surah_name_ar` is filled by the script. Don't hand-edit it; re-run the script after adding new examples.

## ArabicLetter

```ts
interface ArabicLetter {
  arabic: string;
  name_en: string;
  name_ar?: string;
  transliteration?: string;
  condition?: string;        // e.g. "Sakinah preceded by Fathah"
  condition_ar?: string;
  note?: string;
  note_ar?: string;
}
```

## Adding a new example

1. Verify the example exists in the Quran with the exact `surah:ayah` reference (open a Mushaf or use the Quran.com API).
2. Determine the correct `rule_applied`. Match the wording style already in the file (e.g. `Izhar — Noon Sakinah followed by Hamzah (ء)`).
3. Add the example to the rule's `examples` array. Set `surah_name_en` to the standard Latin name; leave `surah_name_ar` empty.
4. Run `node scripts/fetch-surah-names.mjs`. This patches `surah_name_ar` for the new example.
5. Build and load the lesson page. The example should appear with tajweed coloring and audio. If audio is unavailable for that ayah, double-check the surah and ayah numbers.
6. Set `verified: true` only when a reviewer has confirmed accuracy.

## Adding a new rule (advanced)

This is rare; the nine modules cover the standard tajweed curriculum. If you do need to add one:

1. Add the rule object to the appropriate file with all required fields.
2. If it's an entirely new module (not just a new rule in an existing file), create a JSON file in `src/data/content/`, add a TypeScript type for its top-level shape in `src/lib/types.ts`, and add an entry to `learning-path.json`.
3. Add a route under `src/app/learn/[module]/page.tsx` that imports the JSON and renders rule cards.
4. Add an entry in `src/lib/i18n.ts` for any new UI strings.
5. Add the module to `nav-data.tsx` if it should appear in the sidebar.
6. Verify in browser, in EN and AR.

## Things that look like content but aren't

- **Tajweed colors** are defined once in `src/lib/tajweed-colors.ts` and surfaced via CSS classes from the API. Don't change colors per file.
- **Surah names in Arabic** are generated by the script from Quran.com `/chapters`. Don't hand-edit; re-run the script.
- **Verse-end markers** are embedded by the API as `<span class="end">N</span>` inside `text_uthmani_tajweed`. Styling is in CSS.

## Validation

`npx tsc --noEmit` will catch most schema mismatches because the lesson pages import the JSON with the corresponding type. If you add new fields, update the type in `src/lib/types.ts` first.

# Bilingual UI (English / Arabic)

The app translates between English and Arabic across UI chrome, lesson content, common-mistake lists, surah names, weekday letters, and the 404 page. Toggle via the language pill in the sidebar.

## Anatomy

### `src/lib/i18n.ts`

A flat dictionary keyed by string ID, valued by `{ en, ar }`. Keys use dot-notation for grouping:

```ts
"home.startLearning": { en: "Start Learning", ar: "ابدأ التعلّم" },
"mushaf.juz":         { en: "Juz", ar: "الجزء" },
"weekday.short.0":    { en: "S", ar: "ح" },
```

`useTranslation()` is the hook every component uses:

```tsx
const { t, lang, isAr, dir } = useTranslation();
// t("home.startLearning") -> "Start Learning" | "ابدأ التعلّم"
```

### `src/components/layout/AppProvider.tsx`

Wraps every route with `SettingsProvider` (state) plus `SettingsSync` (a child effect that mirrors `settings.language` to `<html lang>` and `<html dir>`). This is what makes `dir="rtl"` flip globally when the user selects Arabic.

## Two layers of bilingual content

### 1. Chrome (i18n.ts)

UI strings: button labels, navigation, error messages, settings labels, weekday letters, mushaf labels. Anything that isn't tajweed content lives here.

To add a new string, append a new key to `i18n.ts` and reference it via `t(key)`. Don't inline an Arabic string in component JSX; that bypasses the i18n layer.

### 2. Tajweed content (JSON files)

Lesson content lives in `src/data/content/*.json`. For every visible English field there's an optional `_ar` counterpart:

| English field | Arabic counterpart |
|---------------|--------------------|
| `title_en` | `title_ar` (always present) |
| `description` | `description_ar` |
| `common_mistakes[]` | `common_mistakes_ar[]` |
| `rule_applied` | `rule_applied_ar` |
| `surah_name_en` | `surah_name_ar` (auto-filled by the script) |
| `translation` | `translation_ar` (optional; many examples skip it) |
| `condition` | `condition_ar` |
| `note` | `note_ar` |
| `introduction` | `introduction_ar` |
| `definition` | `definition_ar` |

Components read either the `_ar` field or fall back to English based on the `useTranslation().isAr` flag.

## How surah names get their Arabic

`scripts/fetch-surah-names.mjs` is a one-shot dev script:

1. Pulls `https://api.quran.com/api/v4/chapters` (114 surahs with `name_arabic`).
2. Writes `src/data/content/surah-index.json` (committed, used as offline fallback).
3. Walks every example across the rule JSON files. For each example with `surah_name_en` but no `surah_name_ar`, it patches `surah_name_ar` from the surah's `name_arabic`.

Run it whenever you add an example referencing a new surah, or whenever the `/chapters` API gets new data:

```bash
node scripts/fetch-surah-names.mjs
```

Commit the resulting JSON changes.

## Hydration-safe localStorage

`useSettings()` reads from localStorage after mount. Server-rendered HTML uses the default settings (English, no bookmarks); the client hydrates and re-renders with the actual settings.

To avoid React hydration warnings on UI driven by localStorage (like the bookmark icon's `fill`), use the `mounted` pattern:

```tsx
const [mounted, setMounted] = useState(false);
useEffect(() => setMounted(true), []);
const isBookmarked = mounted && bookmarks.includes(page);
```

Without this, the server renders `fill="none"` while the client renders `fill="currentColor"` and React complains.

## RTL handling

- `dir="rtl"` and `lang="ar"` are set on `<html>` by `SettingsSync`.
- All Arabic text is wrapped in `<ArabicText>` or `<TajweedText>`. Both set `dir="rtl"` and `lang="ar"` themselves regardless of the surrounding direction.
- Use Tailwind logical properties (`ms-*`, `me-*`, `ps-*`, `pe-*`, `border-s`, `border-e`) so margins and borders flip correctly under RTL.
- For arrows and chevrons, prefer SVGs that carry the geometry (`<polyline points="15 18 9 12 15 6"/>` for a back chevron). The earlier version of the app used `←` and `→` glyphs, which got muddled across direction; chevrons are clearer.

## Translation guidelines

When adding an Arabic translation:

- **Match the register of existing Arabic.** Look at `arabic_definition`, `mnemonic_ar`, and `title_ar` in the same file. They set the tone (formal, scholarly).
- **Use established Tajweed terminology.** Izhar (إظهار), Idgham (إدغام), Iqlab (إقلاب), Ikhfaa (إخفاء), Ghunnah (غنّة), Qalqalah (قلقلة), Madd (مدّ), Tafkheem (تفخيم), Tarqeeq (ترقيق). Don't paraphrase; these are technical terms.
- **Don't over-translate.** Quranic verses (`arabic`) and `arabic_definition` stay in their canonical form. The translations you add are for UI chrome and rule explanations only.
- **When unsure, leave empty.** The UI falls back to English; an empty field is better than a guess.

## Testing

After any i18n change, do a complete pass in both languages and dark mode:

1. Visit `/`, `/learn`, every `/learn/<id>` (9 modules), `/practice`, `/progress`, `/settings`, `/this-doesnt-exist` (404), and `/mushaf` plus a few mushaf pages.
2. For each, toggle the language pill. Confirm there's no English in Arabic mode and vice versa.
3. Confirm `<html dir>` flips with the toggle.
4. Run `node scripts/verify-mushaf.mjs` for the Mushaf flow.
